# PARA Architecture Description

This document describes the intended architecture of PARA:
**Path-Accountable Proof Trees for Software Architecture Relation Reasoning**.
It is written as a detailed system-architecture specification that can be used
to guide diagram generation, manual redrawing, or method-section refinement.

The central message is:

> PARA transforms software architecture relation reasoning from answer
> generation into proof-tree construction. Replaceable strategy sources propose
> typed path programs; the symbolic verifier admits only valid run-local rules;
> the bounded proof engine returns either SUPPORTED with a finite proof tree or
> INCONCLUSIVE as a review obligation.

The agent is part of the strategy proposal layer. It is not the final reasoning
authority. All candidates, whether generated by the agent, deterministic search,
GraphRAG, one-shot LLMs, or ILP calibration, must pass the same symbolic
admission and proof-status contract.

## Drawing Priorities

The architecture drawing should make three ideas visually unavoidable:

1. **The symbolic verifier is the authority boundary.** Strategy sources can
   propose candidates, but the verifier decides what becomes an admitted
   run-local rule. This should be the strongest visual gate in the drawing.
2. **The output is a proof-status contract, not a label.** The right side of
   the drawing should end in two explicit outputs: SUPPORTED with a proof tree,
   and INCONCLUSIVE with a review obligation.
3. **The agent is a replaceable strategy source.** The path-planning agent may
   be highlighted as the adaptive source, but it must remain one source among
   deterministic typed search, GraphRAG, one-shot strategies, and ILP
   calibration. It must not look like the component that directly answers
   queries.

Use visual weight in this order:

```text
1. Symbolic Authority Boundary
2. ProofStatus outputs: SUPPORTED / INCONCLUSIVE
3. ProofStrategy artifact
4. Typed Project Fact Graph G
5. Replaceable strategy sources
6. Audit and replay artifacts
```

If the drawing becomes crowded, remove secondary implementation details before
removing the verifier gate, ProofStrategy artifact, proof engine, or
SUPPORTED/INCONCLUSIVE outputs.

## Overall Architecture

PARA is organized as a left-to-right proof-producing pipeline with four major
regions:

1. **Typed Evidence Layer**
2. **Strategy Proposal Layer**
3. **Symbolic Admission and Proof Layer**
4. **Proof-Status Output and Audit Layer**

The architectural flow is:

```text
Typed project facts
  -> replaceable strategy sources
  -> ProofStrategy artifact
  -> native ingestion
  -> type checking
  -> candidate compilation
  -> symbolic admission
  -> frozen run-local rule library
  -> bounded proof engine
  -> SUPPORTED proof tree or INCONCLUSIVE review obligation
```

The most important architectural boundary is the **Symbolic Authority
Boundary**. Strategy sources may propose candidate path programs, but only the
symbolic verifier can admit them. Only admitted run-local rules enter the proof
engine. This boundary prevents LLM output, retrieval output, or heuristic
candidate generation from being treated as proof.

The recommended visual grammar is:

- strategy-source arrows should point into ProofStrategy, not into final
  answers;
- ProofStrategy should point into native ingestion and symbolic admission;
- rejected candidates should point to verifier diagnostics and optional bounded
  repair, not to proof outputs;
- the bounded proof engine should be the only component that points to
  SUPPORTED or INCONCLUSIVE;
- audit and replay should receive artifacts from outputs and rule libraries,
  not act as a decision-making component.

## Typed Evidence Layer

The Typed Evidence Layer converts project artifacts into a normalized typed
fact graph.

The central object is:

**Typed Project Fact Graph G**

This graph contains typed relational facts with stable entity identifiers.
Typical Java architecture facts include:

- classes
- methods
- packages
- imports
- calls
- inheritance
- containment
- method names
- method arities
- signatures
- annotations
- project-specific policy facts

Engineering and package-level workflows can add other typed facts, such as:

- manifests
- headers
- build metadata
- CMake relations
- package dependencies
- link/runtime validation outcomes

The evidence layer can be supplied by multiple extractors:

**Java Extractors**

- extract class, method, package, import, call, inheritance, containment, and
  signature facts;
- expose architecture-policy labels when available;
- normalize entities into stable typed identifiers.

**Engineering Extractors**

- extract manifest, header, build, and package-dependency facts;
- expose file/package relations for package-slicing and deployment workflows;
- make non-Java evidence available through the same typed relation interface.

**External Validators**

- run build, link, runtime, workload, or stress-test checks;
- validate whether a deployment slice or engineering candidate behaves as
  expected;
- promote an evidence-backed candidate only after external validation succeeds.

The proof core is intentionally insulated from extractor details. Once a fact
has a relation name, typed arguments, and ground tuples, it can participate in
path programs, symbolic admission, and proof contracts.

## Strategy Proposal Layer

The Strategy Proposal Layer contains replaceable sources of candidate
strategies. These sources do not produce final answers. They propose typed path
programs, constraints, and bounds that must be verified downstream.

The layer contains several strategy sources:

### Verifier-Governed Path-Planning Agent

The path-planning agent is the adaptive strategy source. Its role is to propose
executable ProofStrategy artifacts, not to decide whether a query is supported.

The agent contains the following internal components:

- **Intent Profile**: captures the target relation, expected argument types,
  available predicate families, and task context.
- **Agent Memory**: stores reusable strategy fragments, previous accepted
  paths, verifier diagnostics, and domain-specific path patterns.
- **Planner / Refiner**: emits typed path programs and can revise them under
  symbolic feedback.
- **Reflection Trace**: records why a strategy was proposed, why a candidate was
  rejected or preserved, and how verifier feedback affected later proposals.

The agent should be described as:

> a verifier-governed path-strategy proposer.

It should not be described as:

> an autonomous final reasoner or answer generator.

### Deterministic Typed Search

Deterministic typed search proposes candidate paths through local type
compatibility, schema constraints, and simple typed enumeration. It provides a
strong non-LLM anchor for simple structural relations.

### GraphRAG Retrieval

GraphRAG retrieves relevant neighborhoods, paths, or graph evidence that can be
normalized into candidate path strategies. It is useful when graph retrieval
can identify meaningful evidence, but retrieved context is not itself a proof.
The retrieved candidate must still pass symbolic admission.

### One-Shot LLM Strategy

One-shot LLM strategy generation tests whether a model can emit an executable
ProofStrategy without memory, verifier feedback, or repair. These candidates
are useful for calibration but are not trusted unless admitted by the verifier.

### Popper / ILP Calibration

Popper or another ILP system is used as a matched symbolic-search calibration.
It receives the same frozen train split, background facts, held-out split, and
typed predicate bias as the corresponding PARA run. It does not receive
PARA-generated path-program hints or a candidate path family.

This source establishes an important comparison:

- if ILP solves a simple target, pure symbolic search is sufficient there;
- if ILP times out on a compositional target under a fixed typed bias and
  budget, the boundary is search rather than expressiveness;
- a path-hinted ILP variant would be a different interface, because the key
  path family would already have been supplied by PARA-like guidance.

## ProofStrategy Artifact

All strategy sources emit or are normalized into a central artifact:

**ProofStrategy**

The ProofStrategy artifact is a structured, executable strategy object. It is
not a natural-language rationale and not a final answer.

A ProofStrategy includes:

- target relation;
- argument types;
- typed path programs;
- search bounds;
- semantic constraints;
- required predicate families;
- audit requirements;
- optional memory references;
- optional verifier-diagnostic references;
- optional reflection trace metadata.

PARA uses **native ProofStrategy ingestion**. This means the system reads
ProofStrategy objects directly rather than treating them as informal guidance
that must be adapted into a separate learner input. Native ingestion makes the
strategy a first-class replayable artifact.

This design supports several important properties:

- strategy sources are comparable because they share one artifact interface;
- downstream verification is source-independent;
- path programs, constraints, bounds, and trace metadata are preserved;
- reviewers can replay the strategy-to-proof pipeline from stored artifacts.

## Symbolic Admission and Proof Layer

The Symbolic Admission and Proof Layer is the authority-bearing part of PARA.
It determines which candidate strategies become run-local rules and which
queries receive proof-backed support.

This layer should be visually or conceptually enclosed by the:

**Symbolic Authority Boundary**

Inside this boundary are the following stages.

### Native ProofStrategy Ingestion

The system ingests the ProofStrategy artifact and normalizes it into executable
internal objects.

Inputs:

- ProofStrategy artifact;
- target relation schema;
- typed fact graph;
- train examples;
- configured bounds.

Outputs:

- normalized path programs;
- typed candidate templates;
- preserved trace metadata.

### Type Checking

Type checking rejects illegal paths before execution.

It verifies that:

- relation names exist in the fact graph schema;
- argument types match across adjacent path atoms;
- target relation arity and argument types are respected;
- attribute constraints are well typed;
- path variables are consistently bound.

Ill-typed paths are rejected and recorded as verifier diagnostics.

### Path Grounding and Candidate Compilation

Legal path programs are grounded against the typed fact graph and compiled into
Horn-style candidate rules.

This stage converts:

```text
typed executable path programs
```

into:

```text
candidate rules over typed predicates
```

The generated candidates remain provisional. They are not yet admitted and
cannot answer held-out queries.

### Train-Gate Symbolic Admission

The symbolic verifier evaluates candidate rules against the frozen train split.

Inputs:

- candidate rules;
- train examples;
- typed fact graph;
- verifier thresholds;
- target-specific constraints.

Possible outcomes:

- accepted rule;
- rejected candidate;
- verifier diagnostic;
- repair obligation for the strategy source.

Accepted rules are stored in the:

**Frozen Run-Local Rule Library**

Rejected candidates are not silently discarded. Their diagnostics can be used
by the path-planning agent or by developers to understand why the candidate
failed.

### Verifier Diagnostic and Bounded Repair

Verifier diagnostics can flow back to the strategy proposal layer.

Examples of diagnostic feedback include:

- ill-typed path;
- missing predicate family;
- low train precision;
- low train recall;
- unsupported intermediate relation;
- over-specific or under-constrained candidate;
- missing signature constraint;
- missing package/class/method bridge.

The feedback path supports bounded repair:

```text
verifier diagnostic -> strategy refiner -> revised ProofStrategy
```

This feedback does not bypass symbolic admission. A repaired strategy must
return through native ingestion, type checking, candidate compilation, and
symbolic admission.

### Frozen Run-Local Rule Library

Only admitted rules enter the frozen run-local rule library.

This library is:

- specific to a run;
- generated from accepted candidates only;
- separated from held-out labels;
- replayable from stored strategy and split artifacts.

The proof engine uses this library at query time.

### Bounded Proof Engine

The bounded proof engine answers held-out architecture queries using:

- typed fact graph G;
- frozen run-local rule library;
- held-out query q;
- configured proof bounds.

The proof engine does not receive held-out labels during proof search.

For each query, the proof engine attempts to construct a finite derivation. If
support is found, it returns a proof tree. If no bounded support can be shown,
it returns INCONCLUSIVE metadata.

## Proof-Status Output and Audit Layer

The output layer exposes PARA's decision interface.

The output is a:

**Proof-Status Contract**

There are two primary statuses.

### SUPPORTED

SUPPORTED means that PARA found bounded support for the query under the
admitted run-local rule library and typed project facts.

The output includes:

- target query;
- accepted rule used;
- finite proof tree;
- intermediate IDB nodes;
- leaf EDB facts;
- strategy trace reference;
- rule library reference;
- replay metadata.

The proof tree has the following structure:

```text
target query
  -> intermediate derived facts
  -> extracted project facts
```

The leaves of the proof tree must be grounded in extracted EDB facts. The proof
tree is therefore inspectable: reviewers can check which facts were used and
which rules fired.

### INCONCLUSIVE

INCONCLUSIVE means that PARA did not find bounded support under the current
fact graph, admitted rules, and proof bounds.

It is not a negative proof.

The output includes review metadata such as:

- missing support context;
- missing predicate family;
- bound or search gap;
- absent extractor evidence;
- validator requirement;
- possible strategy-refinement request.

INCONCLUSIVE is useful because it prevents the system from silently converting
missing evidence into support. In evaluation, INCONCLUSIVE on a positive
held-out query reduces recall. In deployment, it becomes a structured review
obligation.

## Audit and Replay Artifacts

PARA produces artifacts that support audit and replay.

Important artifacts include:

- ProofStrategy traces;
- verifier diagnostics;
- accepted rule libraries;
- proof-contract samples;
- proof trees;
- EDB leaf checks;
- near-miss stress-test results;
- fact-removal sensitivity checks;
- deterministic replay logs;
- train/held-out split manifests;
- result-summary CSV files.

The audit layer checks whether proof contracts are well formed and whether
SUPPORTED decisions are grounded in real extracted facts.

Typical audit questions include:

- Does every proof-tree leaf correspond to an extracted EDB fact?
- Did every intermediate IDB node follow an admitted rule?
- Does removing a critical fact break support?
- Do near-miss negative queries receive false support?
- Can the proof contract be replayed from stored artifacts?
- Which strategy source produced the admitted path?
- Which verifier diagnostics were generated for rejected candidates?

## Architectural Semantics

PARA's architecture should be understood through the following design
principles.

### Strategy Sources Are Replaceable

The framework does not depend on a single strategy source. Deterministic typed
search, GraphRAG, one-shot LLM strategies, ILP calibration, and the
path-planning agent can all propose candidates.

The shared verifier makes these sources comparable.

### The Agent Proposes, the Verifier Admits

The path-planning agent contributes adaptive path proposal, memory reuse,
diagnostic handling, and bounded repair. It does not decide query support.

The verifier is the authority that admits or rejects candidates.

### Proof Trees Are Operational Evidence

A PARA proof tree is not a theorem-proving certificate of semantic
completeness. It is an operational evidence object produced under configured
bounds from typed project facts and admitted run-local rules.

Its value is reviewability:

- developers can inspect the facts;
- reviewers can replay the derivation;
- unsupported cases remain visible as INCONCLUSIVE;
- missing semantics become extractor or validator obligations.

### INCONCLUSIVE Is a Review Obligation

INCONCLUSIVE should never be interpreted as false. It means the current
evidence, rules, and bounds did not produce support.

Possible next actions include:

- inspect missing facts;
- widen proof bounds;
- add an extractor;
- add a validator;
- request a refined strategy;
- review the query manually.

### Extensibility Lives in the Evidence Layer

The proof core is reusable across ecosystems once typed facts are available.
The ecosystem-specific work belongs in extractors and validators.

For Java architecture reasoning, extractors expose code-graph and policy facts.
For package slicing or static-file workflows, extractors expose manifest,
header, build, and package relations. In both settings, the same proof-status
contract can be used once the typed evidence is available.

## Recommended Visual Encoding

When drawing the architecture, use a clear left-to-right pipeline.

Recommended color semantics:

- Blue: typed facts and evidence;
- Teal or purple: strategy proposal sources;
- Orange: symbolic verifier, admission gate, and proof engine;
- Green: SUPPORTED output and proof tree;
- Gray: INCONCLUSIVE output and review obligation;
- Muted red: rejected candidates and verifier diagnostics.

The Symbolic Authority Boundary should be visually strong. Use a gate, border,
lock, checkmark, or shaded region to make clear that all candidate strategies
must pass symbolic admission.

The path-planning agent may be highlighted slightly as the adaptive strategy
source, but it should not dominate the architecture. It should remain one
replaceable source among several.

Recommended layout:

```text
left: typed evidence and extractors
upper middle: replaceable strategy sources
center: ProofStrategy artifact
center-right: Symbolic Authority Boundary
right: proof-status outputs
bottom: audit and replay artifacts
```

The Symbolic Authority Boundary should contain, at minimum:

- Native Ingestion
- Type Check
- Candidate Compilation
- Symbolic Admission
- Frozen Rule Library
- Bounded Proof Engine

The ProofStatus output region should contain, at minimum:

- SUPPORTED + Proof Tree
- INCONCLUSIVE + Review Obligation
- Audit / Replay

## Statements To Avoid

Do not describe PARA as if:

- the agent directly produces final answers;
- natural-language rationales are proofs;
- INCONCLUSIVE means false;
- proof trees imply semantic completeness;
- the framework universally outperforms ILP, GraphRAG, or deterministic search;
- static-file package reasoning is universal without typed ecosystem facts;
- every held-out query is fully materialized as an evidence-rich proof contract
  by default.

Use the following safer architectural statement instead:

> PARA provides a proof-producing architecture-relation interface. Strategy
> sources propose typed path programs; symbolic admission determines which
> candidates become run-local rules; bounded proof search returns SUPPORTED
> proof trees or INCONCLUSIVE review obligations.
